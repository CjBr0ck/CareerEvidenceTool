# Career Evidence Tool — бриф продукту

Статус: фаза брифу й дизайну. Коду продукту ще немає (в репо лише статична
сторінка-заглушка). Стек навмисно не обрано.

Структура репо й статус кожного розділу — в [README.md](README.md).
Цей файл описує **що ми робимо і чому**; README — **де що лежить**.

## 1. Суть

Система, що збирає робочі досягнення з доказами **в момент, коли вони стаються**.

Фундамент — збір і структурування матеріалу. Виходи (самооцінка для перформанс-рев'ю,
буллети для CV, заготовка кейсу для портфоліо) — **змінні насадки** на цей фундамент,
а не окремі продукти.

**Мета:** одне джерело правди про свою роботу замість реконструкції по пам'яті
напередодні рев'ю. Зібрав матеріал раз — дістав у потрібному форматі, коли настав момент.

## 2. Аудиторія

Фахівці з досвідом — дизайнери, розробники, PM — у командах з регулярними циклами
перформанс-рев'ю. **Не джуни.** Це впливає на тон: без повчань, без гейміфікації,
без пояснень «що таке досягнення».

## 3. Платформа

Веб, desktop-first (логування переважно з робочого місця), далі адаптив до мобільного.

## 4. Рішення, які вже прийнято

| Питання | Рішення |
|---|---|
| Стек | **Не обрано.** Зараз бриф, дизайн, прототипи. Обираємо пізніше, під вимоги. |
| Обсяг MVP | **Тільки збір і перегляд.** Швидке додавання запису з доказом, теги, пошук/фільтр, експорт. Фундамент доводимо до відмінного стану; насадки — наступна фаза. |
| Модель запису | **Гібрид з налаштовуваною обов'язковістю.** Мінімум обов'язкового за замовчуванням, решта полів опційні — але сама «обов'язковість» має бути конфігурованою, щоб можна було протестувати систему й адаптувати. |
| AI | **Не в MVP**, але закладаємо архітектурно. Виходи спершу детерміновані (шаблон + ручне редагування). |
| Ціль | Продукт на ринок, **без монетизації на старті** — апка безкоштовна. |
| Зберігання | **Чисто локально, у користувача.** Дані не залишають його пристрій. Розширимо (синк/хмара) лише якщо буде потреба; безпека даних користувача — пріоритет. |
| Мова | **UA + EN з самого старту** (i18n закладаємо на кожному екрані). |

## 5. Модель запису (evidence unit)

Обов'язкове за замовчуванням: **заголовок + дата**.

Опційні структурні поля: ситуація, дія, результат, метрика, теги, посилання-докази.

Набір обов'язкових полів має бути **конфігурованим** — це свідоме архітектурне рішення,
щоб перевірити на практиці, який поріг входу дає найкращий баланс між регулярністю
логування та якістю матеріалу.

## 6. Типи доказів

Усі чотири підтримуються:

1. **Посилання (URL)** — PR, тікет, Figma-файл, дока, дешборд.
2. **Цитати / фідбек текстом** — похвала в Slack, коментар стейкхолдера, витяг із рев'ю.
   Зі згадкою автора й джерела.
3. **Скріншоти / файли** — зображення та вкладення. Найцінніше для дизайнерів.
4. **Числові метрики** — окреме поле зі значенням до/після та одиницею; саме вони
   потім несуть вагу в буллетах.

## 7. Модель регулярності

Ставка на **обидва режими**:

- **Інбокс** — швидке кидання сирцю: мінімум тертя, вставка посилання, гаряча клавіша,
  закрив. Без вимоги одразу структурувати.
- **Рев'ю** — окремий режим розбору: повернутися до накопиченого, збагатити, структурувати,
  протегувати.

Це прямо адресує головну причину провалу таких систем: у момент події немає ресурсу
заповнювати форму, а через тиждень немає що згадувати.

## 8. Критерії успіху

- **Провідний індикатор — регулярність:** записи з'являються щотижня протягом місяця
  без зовнішнього примусу.
- **Підсумковий — якість матеріалу:** у момент рев'ю самооцінка збирається з наявного,
  без реконструкції по пам'яті.

## 9. Відкриті питання

- Скріншоти/файли + чисто локальне зберігання = нетривіальна пара (обсяг, довговічність,
  формат експорту). Вирішується разом із вибором стеку.
- Формат експорту як гарантія «дані твої» — потрібен відкритий і читабельний.
- Як саме конфігурується обов'язковість полів: пресети чи по-полю.

## 10. Робочі принципи

- Фундамент важливіший за насадки. Виходи не мають диктувати модель даних.
- Тертя при захопленні — головний ворог. Кожне поле в інбоксі доводиться захищати.
- Дані користувача не залишають його пристрій без явного рішення користувача.
- Тон продукту — інструмент для професіонала, не коуч.
